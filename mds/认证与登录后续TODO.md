# 认证与登录后续 TODO

## 1. 当前已经完成

当前已经完成的是第一批认证主链路：

- 登录
- 刷新 access token
- 退出登录
- JWT 生成、验签、过期校验
- Redis 登录会话保存
- 新登录顶掉旧登录
- 单元测试验证主流程

这说明基础链路已经能跑通，但后面还有不少必须继续补的工作。

## 2. 第一优先级：把演示账号替换成真实数据库账号

当前代码里还在用：

- [DemoAuthAccountProvider.java](E:\Smart-Medical\Backend\src\main\java\com\neusoft\neu23\neuhospital\auth\service\DemoAuthAccountProvider.java)

这只是为了先把流程跑通。

下一步应改成：

- 从 `sys_user` 查账号
- 从 `sys_user_role` 查角色关系
- 从 `sys_role` 读取角色编码
- 必要时关联 `doctor` 或 `patient`
- 生成标准化 `AuthAccount`

这一层替换完以后，登录就真正接上业务库了。

## 3. 第二优先级：接入 Spring Security 过滤器链

当前控制器能登录，但业务接口还没有统一鉴权入口。

后面应该补：

- `SecurityConfig`
- JWT 认证过滤器
- 白名单接口配置
- 未认证统一返回
- 无权限统一返回

这样之后，前端请求任何受保护接口时，都会自动走：

1. 取请求头 token
2. 验签
3. 查 Redis session
4. 构造登录上下文
5. 放行到业务接口

## 4. 第三优先级：补用户上下文能力

后面建议增加统一用户上下文对象，例如：

- 当前用户 ID
- 当前用户名
- 当前 userType
- 当前 role
- 当前 bizId
- 当前 sessionId

这样 controller 或 service 里就不用每次手动解析 token。

## 5. 第四优先级：补权限体系

现在只有“能不能登录”，还没有“登录后能干什么”的完整控制。

后面要继续补：

- 角色编码设计
- 权限编码设计
- 菜单权限
- 按钮权限
- 接口权限
- 数据权限

尤其是管理端和医生端，后面基本都离不开权限控制。

## 6. 第五优先级：完善刷新策略

当前第一版刷新逻辑是：

- refresh token 有效时，重新签发新的 access token

后续可以继续增强成：

- 刷新时轮换新的 refresh token
- 旧 refresh token 立即失效
- 记录刷新次数或最后刷新时间
- 对异常刷新行为做风控

如果后期要做更严格的安全控制，建议采用 refresh token 轮换。

## 7. 第六优先级：完善登录方式

目前主流程是按账号密码模型设计的。

后面要逐步扩展：

- 患者端手机号验证码登录
- 微信小程序登录
- 医生端账号密码登录
- 管理端账号密码登录

推荐顺序：

1. 先把账号密码主链路和权限跑稳
2. 再加患者端手机号验证码
3. 最后接微信小程序静默登录

## 8. 第七优先级：接微信小程序身份

患者端如果是微信小程序，后续建议接入：

- `openid`
- `unionid`
- 小程序登录态换取
- 本地用户和微信身份绑定

这样之后患者端体验会更顺，不需要频繁手输账号密码。

但这一步建议放在基础认证稳定之后再做，不要一开始把复杂度拉满。

## 9. 第八优先级：补审计和风控

后续建议记录这些信息：

- 登录时间
- 登录 IP
- 设备信息
- 退出时间
- 刷新 token 时间
- 失败登录次数
- 管理员强制下线记录

这部分对医疗系统尤其重要，因为很多操作都需要追踪。

## 10. 第九优先级：补接口测试和集成测试

当前已经有单元测试，但还不够。

后续建议继续补：

- 登录接口集成测试
- 刷新接口集成测试
- 退出接口集成测试
- Redis 真实环境测试
- 异常 token 测试
- 多端重复登录测试

尤其是“新登录顶掉旧登录”“退出后旧 token 失效”这类场景，要通过集成测试固化下来。

## 11. 第十优先级：把三端账号模型彻底收敛

后面最好统一约束：

- 所有能登录的人，先都落到 `sys_user`
- 角色走 `sys_role`
- 用户角色关系走 `sys_user_role`
- 患者业务资料放 `patient`
- 医生业务资料放 `doctor`
- 管理类人员先走 `sys_user` + 管理角色

这样认证层就不会和业务表强耦合。

## 12. 建议的近期开发顺序

最建议的顺序是：

1. 用数据库表替换演示账号提供器
2. 接 Spring Security 过滤器链
3. 把登录用户上下文打通
4. 做接口权限校验
5. 完善患者端 / 医生端 / 管理端角色模型
6. 接患者端手机号或微信登录
7. 增强审计、风控和会话治理

## 13. 一句话总结

当前这版已经不是“空谈设计”，而是主认证流程已经能工作。

接下来最关键的，不是继续堆更多概念，而是把这条认证链真正接到：

- 真实数据库账号
- 真实接口鉴权
- 真实三端权限

这样后面再接 AI、Agent、MCP、Python 服务时，底座才是稳的。
