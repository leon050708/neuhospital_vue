# 智慧云脑诊疗 Agent 平台数据库表设计（第一批核心表）

> 用途：用于组内对齐第一阶段认证与身份核心表。  
> 原则：恢复到“比极简方案更规范、但仍可控”的版本，优先解决三端登录、角色扩展和科室区分。

---

## 1. 当前采用的核心方案

第一阶段先落 6 张核心表：

```text
1. sys_user        # 统一登录账户表
2. sys_role        # 角色表
3. sys_user_role   # 用户角色关联表
4. patient         # 患者档案表
5. department      # 科室表
6. doctor          # 医生档案表
```

这 6 张表已经足够支撑：

```text
患者端登录
医生端登录
管理端登录
一个用户可扩展到多个角色
医生归属不同科室
后续按科室做数据权限隔离
JWT 中 userType / role / bizId 的身份承载
```

---

## 2. 为什么恢复 `sys_role / sys_user_role`

极简方案里把角色直接塞进 `sys_user.role_code`，适合快速起步，但你们现在已经明确有一个关键需求：

- 不同科室要区分

这说明后面权限不会只停留在“患者 / 医生 / 管理员”三类。

后续很容易出现：

- 医生
- 科主任
- 影像医生
- 检验医生
- 挂号收费员
- 系统管理员

如果只靠 `sys_user` 一张表放一个 `role_code`，后面会遇到两个问题：

- 一个用户难以兼任多个角色
- 角色扩展和权限配置会越来越别扭

所以当前恢复为：

- `sys_user`：账号是谁
- `sys_role`：系统里有哪些角色
- `sys_user_role`：某个账号拥有哪些角色

这比极简方案更适合后续演进。

---

## 3. 先说明一件事：区分科室主要不是靠角色表

这个点要讲清楚，不然容易误解。

“不同科室”主要靠的是：

- `doctor.department_id`

也就是说：

- 角色表解决“你是什么身份”
- 科室字段解决“你属于哪个科室”
- 后续数据权限解决“你能看自己科室还是全院数据”

例如：

- `role = DOCTOR`
- `department_id = 心内科`

和：

- `role = DOCTOR`
- `department_id = 神经内科`

这两个医生角色相同，但所属科室不同。

所以正确建模应该是：

```text
角色 = 身份标签
科室 = 组织归属
数据权限 = 可查看的数据范围
```

## 4. 第一阶段统一设计约定

### 4.1 主键约定

- 主键统一使用 `BIGINT`
- 字段名统一为 `id`
- 当前建议 PostgreSQL 使用自增主键

### 4.2 审计字段约定

核心表建议统一包含：

- `created_at`
- `updated_at`
- `created_by`
- `updated_by`
- `deleted`

### 4.3 状态字段约定

状态字段建议统一用明确枚举值：

- 用户状态：`ENABLED` / `DISABLED`
- 角色状态：`ENABLED` / `DISABLED`
- 患者状态：`ENABLED` / `DISABLED`
- 医生状态：`ENABLED` / `DISABLED`
- 科室状态：`ENABLED` / `DISABLED`

---

## 5. `sys_user` 是什么

`sys_user` 是统一登录账户表，不是患者明细表，也不是医生明细表。

它只负责：

```text
谁能登录
登录名是什么
密码怎么存
这个账号属于患者、医生还是管理端
这个账号绑定的是哪条业务身份
```

建议这样理解：

```text
sys_user = 登录壳子
patient  = 患者业务档案
doctor   = 医生业务档案
```

例如：

- 患者账号：`user_type = PATIENT`，`biz_id = patient.id`
- 医生账号：`user_type = DOCTOR`，`biz_id = doctor.id`
- 管理端账号：`user_type = MANAGEMENT`，`biz_id` 可为空

## 6. 第一阶段 6 张核心表设计

## 6.1 `sys_user` 登录用户表

**作用：**

- 统一管理登录账号
- 支撑 JWT 登录认证
- 区分患者端、医生端、管理端
- 通过 `biz_id` 绑定患者或医生业务身份

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| id | bigint | 是 | 主键 |
| username | varchar(64) | 是 | 登录账号，建议唯一 |
| password_hash | varchar(255) | 是 | 密码密文 |
| user_type | varchar(32) | 是 | 用户类型：PATIENT / DOCTOR / MANAGEMENT |
| biz_id | bigint | 否 | 业务身份 ID，患者时关联 patient.id，医生时关联 doctor.id |
| real_name | varchar(64) | 是 | 真实姓名 |
| phone | varchar(32) | 否 | 手机号 |
| email | varchar(128) | 否 | 邮箱 |
| status | varchar(32) | 是 | 账号状态：ENABLED / DISABLED |
| last_login_at | timestamp | 否 | 最后登录时间 |
| created_at | timestamp | 是 | 创建时间 |
| updated_at | timestamp | 是 | 更新时间 |
| created_by | bigint | 否 | 创建人 |
| updated_by | bigint | 否 | 更新人 |
| deleted | boolean | 是 | 逻辑删除标记 |

**建议约束：**

- `username` 唯一索引
- `phone` 可选唯一索引
- `(user_type, biz_id)` 普通索引

## 7. 第一阶段表关系

```text
sys_user
  ├─ user_type = PATIENT    -> biz_id 关联 patient.id
  ├─ user_type = DOCTOR     -> biz_id 关联 doctor.id
  └─ user_type = MANAGEMENT -> biz_id 可为空

sys_user
  └─ 通过 sys_user_role 关联 sys_role

doctor
  └─ 通过 department_id 关联 department
```

这里有两个核心维度：

- 角色维度：通过 `sys_role`
- 科室维度：通过 `doctor.department_id`

---

## 8. 三端账号建议

### 患者端

- `user_type = PATIENT`
- 常见角色：`PATIENT`
- `biz_id = patient.id`

### 医生端

- `user_type = DOCTOR`
- 常见角色：`DOCTOR`
- 可扩展角色：`DEPT_ADMIN`、`IMAGE_DOCTOR`、`INSPECTION_DOCTOR`
- `biz_id = doctor.id`

### 管理端

- `user_type = MANAGEMENT`
- 常见角色：`REGISTRATION_CLERK`、`CASHIER`、`ADMIN`
- `biz_id` 可为空

---

## 9. 为什么这版更适合“不同科室”

因为后续区分不同科室通常不是简单写成：

- 心内科医生角色
- 神经内科医生角色

那样角色会爆炸。

更合理的是：

- 角色统一还是 `DOCTOR`
- 科室靠 `doctor.department_id`
- 后续数据范围控制为：
  - 只能看本人
  - 只能看本科室
  - 可以看全院

这就是为什么要把“角色”和“科室”分开建模。

---

## 10. 当前结论

当前推荐第一阶段按下面这套推进：

```text
sys_user + sys_role + sys_user_role + patient + department + doctor
```

这套方案比极简版更规范，也比一步到位全量业务表更可控。

它足够支撑：

- 三端登录
- 不同角色扩展
- 医生归属不同科室
- 后续科室级数据权限
