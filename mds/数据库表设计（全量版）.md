# 智慧云脑诊疗 Agent 平台数据库表设计（全量版）

> 用途：统一当前项目数据库设计口径，合并第一批认证身份核心表与后续业务表、AI 表、影像表。  
> 技术前提：`Spring Boot 3.2.x + MyBatis-Plus + PostgreSQL + Redis + Kafka + MinIO + Spring AI`

---

## 1. 文档范围

本版数据库设计覆盖以下 8 个域：

```text
1. 认证与身份域
2. 系统治理域
3. 挂号与排班域
4. 门诊接诊与病历域
5. 检查 / 检验 / 处置域
6. 处方、药房与缴费域
7. AI 能力域
8. 文件与医学影像域
```

说明：

- 本文以“能支撑当前课程项目完整闭环”为目标，不追求一步到位的超细化 HIS 建模。
- 对于特别重的能力，例如医保结算、住院、病案首页国标、复杂库存批号管理，当前不纳入本版。
- Redis、Kafka、MinIO 是中间件，不在本文设计成关系表，但会体现在字段关联和业务说明里。

## 2. 统一设计约定

### 2.1 主键约定

- 主键统一使用 `BIGINT`
- 字段名统一为 `id`
- PostgreSQL 推荐使用 `BIGSERIAL` 或 `GENERATED BY DEFAULT AS IDENTITY`

### 2.2 通用审计字段

除明确说明外，所有业务主表默认包含以下字段：

| 字段名 | 类型 | 说明 |
|---|---|---|
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |
| created_by | bigint | 创建人 |
| updated_by | bigint | 更新人 |
| deleted | boolean | 逻辑删除标记，默认 `false` |

### 2.3 常用状态字段约定

常见状态值建议统一采用明确枚举：

```text
ENABLED / DISABLED
DRAFT / CONFIRMED
PENDING / PROCESSING / SUCCESS / FAILED
CREATED / PAID / CANCELLED / COMPLETED
UNPAID / PAID / PARTIAL_REFUND / REFUNDED
UNCONFIRMED / CONFIRMED / REJECTED
```

### 2.4 编号字段约定

凡是面向业务流转、日志追踪和人工检索的主业务单据，建议都带业务编号：

- `patient_no`
- `doctor_no`
- `registration_no`
- `record_no`
- `request_no`
- `prescription_no`
- `order_no`
- `task_no`

### 2.5 索引设计原则

- 所有外键关联字段都应建立普通索引
- 所有业务编号字段都应建立唯一索引
- 高频查询组合建议建立联合索引，例如：
  - `(doctor_id, schedule_date, time_slot)`
  - `(patient_id, visit_date)`
  - `(session_id, sequence_no)`
  - `(task_id, status)`

## 4. 认证与身份域

## 4.1 `sys_user` 统一登录账户表

**作用：**

- 统一管理患者端、医生端、管理端账号
- 支撑 JWT 登录认证
- 通过 `biz_id` 绑定患者或医生业务身份

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| id | bigint | 是 | 主键 |
| username | varchar(64) | 是 | 登录账号，唯一 |
| password_hash | varchar(255) | 是 | 密码密文 |
| user_type | varchar(32) | 是 | `PATIENT / DOCTOR / MANAGEMENT` |
| biz_id | bigint | 否 | 业务身份 ID，患者对应 `patient.id`，医生对应 `doctor.id` |
| real_name | varchar(64) | 是 | 真实姓名 |
| phone | varchar(32) | 否 | 手机号 |
| email | varchar(128) | 否 | 邮箱 |
| status | varchar(32) | 是 | `ENABLED / DISABLED` |
| last_login_at | timestamp | 否 | 最后登录时间 |

**建议约束：**

- `username` 唯一索引
- `phone` 可选唯一索引
- `(user_type, biz_id)` 普通索引

## 12. 关键表关系说明

### 12.1 认证身份关系

```text
sys_user
  ├─ user_type = PATIENT    -> biz_id 关联 patient.id
  ├─ user_type = DOCTOR     -> biz_id 关联 doctor.id
  └─ user_type = MANAGEMENT -> biz_id 可为空

sys_user
  └─ 通过 sys_user_role 关联 sys_role

sys_role
  └─ 通过 sys_role_permission 关联 sys_permission
```

### 12.2 挂号就诊主链路

```text
doctor_schedule -> registration -> medical_record
registration    -> payment_order
medical_record  -> medical_diagnosis
medical_record  -> check_request / inspection_request / disposal_request / prescription
```

### 12.3 检查检验链路

```text
check_request      -> check_result
inspection_request -> inspection_result -> inspection_result_item
disposal_request   -> 执行状态流转
```

### 12.4 处方收费链路

```text
prescription -> prescription_item
prescription -> drug_dispense_record
payment_order -> payment_item -> 各类业务单据
payment_order -> refund_record
```

### 12.5 AI 与影像链路

```text
ai_chat_session      -> ai_chat_message
ai_chat_session      -> ai_triage_result
medical_record       -> ai_medical_record_draft
medical_record       -> ai_diagnosis_advice
knowledge_document   -> knowledge_chunk -> knowledge_vector
file_record          -> ct_image_file -> ct_analysis_task -> ct_analysis_result
```

## 14. 当前结论

当前数据库推荐采用：

```text
认证与身份底座
+ 系统治理扩展
+ 门诊主业务闭环
+ AI 会话与知识库
+ CT 影像任务与结果
```

这套设计能覆盖：

- 三端登录
- RBAC 权限扩展
- 科室与医生归属
- 排班、挂号、接诊、病历、处方、缴费
- 检查检验申请与结果
- AI 问诊、导诊、病历草稿、辅助诊断
